● Assessment: Snake Game Specification

  Grounding Check

  Several terms need resolution before implementation (#ground):

  - "Middle of an 8x8 board" — an 8x8 grid has no single center cell.
  Candidates: (3,3), (3,4), (4,3), (4,4). A size-3 snake moving right occupying
  which cells? Likely cells (2,3), (3,3), (4,3) with head at (4,3) — but
  "middle" on an even grid is ambiguous. (#ground)
  - "Speed increases after every 10 points" — what's the initial speed? What's
  the increment? Does it cap? (#ground)
  - "Occasionally a star spawns" — what probability, checked when? Every turn?
  Only when no star is present? (#ground)
  - "20 turns" — a "turn" is one snake movement tick, correct? (#ground)
  - "Points" — 1 per apple? 1 per growth unit? Does the star give 1 point or 3?
  (#ground)

  Tensions

  - Key stacking + self-collision rule: If moving right and user rapidly presses
   s then a (down, then left), after the first tick the snake faces down — then
  a (left) is valid. But if evaluated at queue time against current direction, a
   would be rejected (opposite of right). The rule must validate each queued
  input against the direction the snake would have after processing prior queue 
  entries, not the current direction. This is the subtlest bug vector in the
  spec.
  - Board size vs. game viability: 8x8 = 64 cells. A snake growing by 1 (apple)
  or 3 (star) on a 64-cell board runs out of space quickly. Max theoretical
  snake length = 64. Food placement must avoid occupied cells — as the snake
  fills the board, valid spawn locations shrink to zero. What happens when the
  board is full? (#negative-space)
  - Star + apple spawn conflicts: Both require empty cells. Can they occupy the
  same cell? Presumably not. But neither the interaction nor priority is
  specified. (#negative-space)

  What's Absent (#negative-space)

  - Win condition — is filling the entire board a win, or does the game only end
   on wall/self collision?
  - Wall collision — never mentioned. Does the snake wrap, die, or is the
  boundary implicit? This is a critical omission.
  - Apple point value — implied 1 but unstated.
  - Star point value — unstated.
  - Initial speed — unstated (in ticks/sec or ms/tick).
  - Speed increment formula — unstated.
  - Speed cap — unstated. Without one, the game eventually moves faster than
  human reaction time, then faster than the key repeat rate, then faster than
  the display refresh.
  - What "game end" means — only self-collision? Wall collision? Board-full?
  - Visual design — cell size, colors, window dimensions, font for score
  display.
  - Star spawn trigger — probability per tick? Fixed interval? Random window?

  Fractal Analysis

  At macro (game loop): spec is coherent — init, input, update, render, repeat.
  At meso (subsystems): input queuing, spawning logic, collision detection each
  have under-specified edges.
  At micro (cell-level): the 8x8 grid with even dimensions creates ambiguity in
  "middle." (#fractal)

  The meso layer has the most gaps — the macro design is standard, the micro is
  concrete once grounding questions resolve. (#fractal)

  Meta

  Assessing whether assessment is the right mode: the spec has enough ambiguity
  that building now would require dozens of implicit decisions. Assessment is
  correct — but most gaps have obvious default answers (walls kill, apple = 1
  point, etc.). Little remains to assess beyond the items above. (#meta)

  Priorities

  1. Wall behavior — blocks implementation entirely
  2. Key queue validation semantics — blocks correct input handling
  3. Speed parameters (initial, increment, cap) — blocks game feel
  4. Point values (apple, star) — blocks scoring
  5. Star spawn probability — blocks spawning logic
  6. "Middle" resolution — minor, pick a convention

  When these resolve → #=spec or #=code.

  Bug Report

  BUG 1: tick() has no phase guard — zombie snake

  Severity: Medium | game.py:68

  tick() unconditionally processes movement and hardcodes phase="playing" on
  line 119. Calling it on a "lose", "win", or "start" state produces a "playing"
   state — resurrecting a dead snake or auto-starting a game.

  The game loop in main.py:102 prevents this (if state.phase == "playing"), but
  the API contract is broken. Any future caller, test harness, or refactor that
  calls tick() on a non-playing state gets silent corruption. (#deep — layer 1:
  works now; layer 2: fragile to change; layer 3: the function's return type
  promises phase="playing" regardless of input, which is a lie about its
  contract.)

● Review

  Bug (must fix)

  main.py:105-106 — interval computed once, used for cap AND tick loop in same 
  frame

  interval = 1000 // speed(state.score)
  tick_accum += min(dt, interval)

  speed(state.score) is evaluated once at the top. If the score changes mid-loop
   (snake eats apple inside the while loop), the interval used for subsequent
  loop iterations is stale. In practice the min(dt, interval) cap limits to one
  tick per frame, so this currently can't trigger. But the cap depends on the
  staleness not mattering, and the staleness depends on the cap — a circular
  safety argument. If either assumption breaks under refactoring, the other
  silently stops holding. (#challenge — should the interval recompute per
  iteration to make the two properties independently correct?)

  ---
  Design (discuss)

  1. "Speed: 3" is opaque to the player — main.py:53, game.py:27

  The user sees "Speed: 3" through "Speed: 8". What is a "3"? Ticks per second —
   but a player doesn't think in those units. "Level 1" through "Level 6" maps
  to the player's mental model: higher level = harder. Or even just not showing
  the raw number and showing a bar or descriptive word. (#user-lens — (#ground)
  what does "Speed: 3" resolve to in the player's head?)

  2. Queue limit of 2 silently drops the third distinct move — game.py:39

  A player pre-planning 3 moves (DOWN, LEFT, UP in rapid succession at a corner)
   loses their third input silently. They pressed a key, got no feedback, and
  the snake doesn't turn when expected. This is the most likely frustration
  point for a skilled player. (#user-lens)

  3. Star has no visual urgency signal — main.py:49-50

  The star appears, sits there for 20 ticks, and vanishes. The player has no way
   to judge how much time remains. A blinking or dimming star (e.g., faster
  blink as timer drops below 5) would communicate urgency. Without it, the
  star's disappearance feels arbitrary rather than telegraphed. (#user-lens)

  4. Death cause is invisible — main.py:64

  "Game over!" looks the same for wall collision and self collision. The player
  just died — their first question is "what did I hit?" Showing "Hit wall!" vs
  "Hit self!" costs one field on GameState and resolves the user's immediate
  confusion. (#user-lens)

  ---
  Style (noted once)

  Positional GameState construction — game.py:72,94,104,129

  GameState(snake, RIGHT, apple, None, 0, 0, "start") relies on field order.
  GameState(snake=..., direction=..., ...) survives field reordering. The
  dataclass has 7 fields — one swap silently produces wrong state.
