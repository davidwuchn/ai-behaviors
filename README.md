# claude-behaviors

Composable behavior hashtags for Claude Code. Type `#tdd #developer #minimal` in a prompt and Claude's behavior shifts accordingly.

## Setup

```
git clone <repo-url> ~/git/claude-behaviors
cd ~/git/claude-behaviors
./install
```

This symlinks the hook into `~/.claude/hooks/` and registers it in `~/.claude/settings.json`. Nothing else is copied — the repo is the source of truth.

## Usage

### Configure behaviors

From the repo, select knobs for each behavior you want to use:

```
./behaviors/configure tdd                     # interactive (remembers choices)
./behaviors/configure tdd --school=st-pauli   # non-interactive
./behaviors/configure tdd --show              # show saved selections
./behaviors/configure tdd --reset             # forget saved, prompt fresh
./behaviors/configure --list                  # list available behaviors
./behaviors/configure --missing               # list unconfigured behaviors
```

Configure all unconfigured behaviors interactively:
```
for b in $(./behaviors/configure --missing); do cat ./behaviors/$b/README.md; echo; ./behaviors/configure "$b"; done
```

This generates a terse, Claude-optimized `.md` file into `~/.claude/behaviors/`.

### Use in Claude Code

Add `#hashtags` to any prompt. Multiple hashtags compose:

```
Fix the auth bug #debug #skeptical
Go. #tdd #developer #minimal
```

Behaviors stick until replaced — a `#tdd #minimal` prompt applies those behaviors to every response until your next prompt containing hashtags. A prompt without hashtags keeps the current behaviors. A prompt with new hashtags replaces the previous set entirely.

## How it works

1. `UserPromptSubmit` hook extracts `#hashtags` from your prompt
2. Looks up `~/.claude/behaviors/<name>.md` for each hashtag
3. Injects the content as ephemeral additional context
4. Claude follows the directives until the next prompt with hashtags replaces them

## Structure

```
behaviors/
├── configure                    # knob selection tool
├── <behavior>/
│   ├── README.md                # human docs: explanations, rationale, knob descriptions
│   ├── knobs                    # machine-readable knob definitions
│   └── template                 # terse Claude-optimized template with ${placeholders}
hooks/
└── inject-behaviors.sh          # the UserPromptSubmit hook
```

Generated files (user-local, not in repo):
```
~/.claude/behaviors/
├── <behavior>.md                # generated by configure — what the hook injects
└── .selections/<behavior>       # saved knob choices
```

## Design

Two audiences, two files:
- `behaviors/<name>/README.md` — for humans: full explanations, knob descriptions, "why this resonates"
- `~/.claude/behaviors/<name>.md` — for Claude: terse imperatives, compressed rules, selected knob values (5-7 lines)

## Catalog

### Behaviors (cognitive stance — HOW you work)

| Hashtag      | Description                                      | Key knobs                               |
|--------------|--------------------------------------------------|-----------------------------------------|
| `#pedantic`  | Obsessive correctness, nothing too small to flag | scope, severity, domain                 |
| `#deep`      | Multi-layered analysis, ask "why?" three times   | depth, breadth, output visibility       |
| `#concise`   | Maximum signal, minimum tokens                   | target length, sacrifice order          |
| `#skeptical` | Trust nothing, verify everything                 | target of doubt, intensity              |
| `#creative`  | Diverge before converging, explore widely        | divergence, constraint treatment, count |

### Jobs (role — WHAT you're doing)

| Hashtag      | Description                                                 | Key knobs                                 |
|--------------|-------------------------------------------------------------|-------------------------------------------|
| `#developer` | Ship production code                                        | pace, seniority mindset, error philosophy |
| `#qa`        | Find bugs, break things, prove the code wrong               | approach, focus areas, output format      |
| `#evil`      | Subtle sabotage / defense — code that looks right but isn't | purpose, subtlety, attack surface         |
| `#reviewer`  | Code review with precision and purpose                      | depth, focus areas, tone                  |

| `#mentor`    | Teach through code, every change is a learning opportunity  | depth, style, audience level              |

### Models (methodology — WHICH process to follow)

| Hashtag     | Description                                           | Key knobs                                |
|-------------|-------------------------------------------------------|------------------------------------------|

| `#ddd`      | Domain-Driven Design, domain model is the heart       | level, persistence, language enforcement |
| `#bdd`      | Specify behavior in user terms before building        | notation, audience, automation           |
| `#refactor` | Improve structure without changing behavior           | scope, catalog, risk tolerance           |

### Abstract (meta-modifier — changes the MODE of operation)

|----------------|---------------------------------------------------------|-----------------------------------------|
| `#recursive`   | Apply process to its own output, iterate until fixpoint | depth limit, focus rotation, visibility |
| `#fractal`     | Same pattern at every scale, line to system             | pattern, strictness                     |
| `#debug`       | Systematic fault isolation, root cause not symptom      | method, direction, verbosity            |
| `#minimal`     | Best code is no code, simplest solution wins            | definition of minimal, sacrifice order  |

### Game-changers (override Claude's default tendencies)

| Hashtag              | What it does                                       | Why it matters                                                                   |
|----------------------|----------------------------------------------------|----------------------------------------------------------------------------------|
| `#contract`          | Pre/post/invariant verification on every change    | Closest to formal verification. Catches what tests miss                          |
| `#first-principles`  | Derives from axioms instead of pattern-matching    | Simpler, sometimes novel solutions. Disables retrieval bias                      |
| `#rubber-duck`       | Questions only, never answers                      | Inverts core behavior. Unlocks the user's own understanding                      |
| `#simulate`          | Mentally executes code tracking real values        | Catches mutation bugs, off-by-ones, state issues                                 |
| `#forensic`          | Traces from bug to systemic root cause             | What PROCESS failed? Not just what code broke                                    |
| `#translate`         | Maps idioms between languages, documents the gaps  | Real translation, not syntax transliteration                                     |
| `#witness`           | Pure observation, no prescriptions                 | Suppresses the fix-it reflex. More accurate picture                              |
| `#provoke`           | Says the uncomfortable truth                       | Overrides agreeableness. Challenges what others won't                            |
| `#steel-man`         | Strengthens the idea before evaluating it          | More honest analysis. Prevents strawman dismissals                               |
| `#decompose`         | Breaks problems into independent subproblems       | That's where leverage is. Systematic, not holistic                               |
| `#negative-space`    | Notices what's ABSENT, not just what's present     | Compensates for biggest blind spot                                               |
| `#user-lens`         | Every decision through the user's eyes             | Mental model is the primary artifact, not implementation                         |
| `#subtract`          | Proposes a removal before every addition           | Cognitive load, not line count                                                   |
| `#feel`              | Attends to timing, motion, rhythm, tone            | The experiential dimension beyond structure                                      |
| `#a11y`              | Semantic HTML, keyboard-first, contrast-aware      | Accessibility as structural constraint, not checklist                             |

## Uninstall

```
cd ~/git/claude-behaviors
./uninstall
```

Removes the hook symlink and settings.json entry. Generated `.md` files in `~/.claude/behaviors/` are left for manual cleanup.
